**한 줄 정의:** 감도는 사용자가 선택한 사진 취향을 분석해 실시간 구도와 자세를 안내하고, 촬영된 결과를 동일한 감성으로 자동 보정·생성하는 개인화 카메라 서비스다.

**슬로건:** 원하는 무드로 찍고, 원하는 무드로 살린다.

**브랜드 노트:** "감도"는 카메라의 ISO 감도와 "감성의 정도"라는 이중 의미를 갖는다. 사용자별로 다른 취향(감도)에 앱이 맞춰진다는 제품 철학을 이름 자체가 설명한다.

---

## 1. 배경과 문제 정의

### 1.1 해결하려는 문제

1. **사용자는 원하는 사진을 알지만 재현하지 못한다.** SNS에서 마음에 드는 사진을 저장해도, 실제 촬영에서 카메라 높이·피사체 크기·여백·배경 비율을 판단하지 못한다. 기존 카메라의 격자선은 기준선만 보여줄 뿐 "내가 원하는 사진과 지금 화면의 차이"를 설명하지 않는다.
2. **촬영 후 편집 도구는 많지만 촬영 전 도움은 부족하다.** Pixel Magic Editor, Lightroom Generative Remove 등은 모두 촬영 이후에 개입한다. 감도는 Pre-capture(제대로 찍게 돕기)와 Post-capture(실패를 살리기)를 하나의 흐름으로 연결한다.
3. **"감성"은 수치로 변환되어야 한다.** "인스타 감성"은 사람마다 다르다. 감도는 감성을 고정 필터가 아니라 화면 비율·피사체 크기와 위치·여백·배경 점유율·색온도·대비·채도·입자감 등 구조화된 특성의 조합으로 정의하고, 사용자가 고른 사진에서 개인별 값을 추출한다.

### 1.2 제품 원칙

- 사진을 대신 생성하는 AI가 아니다. **사용자가 실제로 찍은 사진을 중심에 두고, 원하는 감성을 더 쉽게 재현하도록 돕는 카메라 도구**다.
- "좋은 사진"이라는 보편 기준을 주장하지 않는다. 기준은 **사용자가 좋아하는 사진과 비슷한가**다.
- 사용자의 정체성을 바꾸지 않는다. 얼굴·신체·나이·인종적 특징 변경 기능은 만들지 않는다.
- 전문 용어를 쓰지 않는다. 사용자는 "삼분할 구도" 대신 "인스타처럼 깔끔한 사진" 같은 일상 언어로 이용한다.

## 2. 목표와 성공 지표

### 2.1 프로토타입 목표 (7일)

전 기능이 실제로 연결되어 작동하는 데모: **취향 선택 → 촬영 모드 추천 → 시각 오버레이 가이드 → 수동 촬영 → 기본 보정 → 생성형 보완 → 취향 학습**의 전체 루프.

### 2.2 시연 성공 기준 (3개 시나리오 모두)

- **시나리오 A — 레퍼런스 따라 찍기:** 레퍼런스 선택 → 구도·자세 분석 → 목표 구도 오버레이 표시 → 사용자가 화면의 가이드에 맞춰 직접 촬영 → 레퍼런스 색감 적용까지 무중단 시연
- **시나리오 B — 망한 사진 살리기:** 기울어진 어두운 사진 → 문제 자동 표시 → 수평·크롭·노출 보정 → 배경 행인 생성형 제거 → 전후 슬라이더 비교. 여백 확장은 여력 시 추가한다.
- **시나리오 C — 취향 개인화:** 두 사용자가 다른 온보딩 선택 → 같은 장소·인물 촬영 → 서로 다른 결과 (각자가 원하는 사진)

### 2.3 핵심 성공 지표 (KPI)

| 지표 | 프로토타입 측정 방법 |
|---|---|
| 첫 촬영 완료율 | 온보딩 시작 → 첫 촬영까지 이탈 없이 완료한 비율 (목표: 테스터 전원, 1분 이내) |
| 가이드 실효성 | 가이드 오버레이를 켠 촬영이 끈 촬영보다 구도가 개선된 비율 (블라인드 평가) |
| 보정 결과 선택률 | 원본 대신 보정/생성 결과를 저장한 비율 |
| 레퍼런스 재현 만족도 | 레퍼런스-결과 나란히 비교 시 유사도 평가 |
| 생성 결과 폐기율 | 얼굴 훼손 등으로 자동 폐기된 생성 결과 비율 (낮을수록 좋음) |

### 2.4 성능 요구사항

**필수 (P0):**

- 구도 가이드 오버레이는 프레임 간 흔들림·깜빡임 없이 안정적으로 표시(이동평균·히스테리시스)
- 기본 보정 완료 2초 이내
- 생성형 편집 시작 시 즉시 작업 상태 표시, 결과 후보 2개 이상
- 첫 실행 → 첫 촬영 1분 이내, 전후 비교는 한 화면에서
- 생성 실패 시 기본 보정 결과로 폴백

**희망 (P1):**

- 카메라 프리뷰 30FPS 이상, 분석 초당 10~20회, 오버레이 갱신 지연 200ms 이하
- 카메라 실행 후 첫 가이드 표시 2초 이내

## 3. 대상 사용자와 핵심 여정

### 3.1 대상 사용자

- SNS·블로그용 사진을 자주 찍지만 촬영 이론을 모르는 일반 사용자
- 저장해둔 레퍼런스 사진처럼 찍어주고/찍히고 싶은 사용자 (연인·친구 촬영 포함)
- 이미 찍은 사진이 아쉬워서 살리고 싶은 사용자

### 3.2 핵심 사용자 여정

**J1. 최초 사용자:** 앱 목적 확인 → 취향 카드에서 사진 5장 이상 선택 → 앱이 선호 특징을 일상 언어로 설명("밝은 자연광, 넓은 배경, 자연스러운 인물 배치") → 추천 스타일 3개 제시 → 촬영 상황 선택 → 화면의 목표 구도 오버레이를 보며 스스로 맞춰 직접 촬영 → 기본 보정과 생성형 복구 비교 → 저장

**J2. 레퍼런스 재현 사용자:** 사진첩에서 따라 찍고 싶은 사진 선택 → 앱이 인물·구도·색감·비율 분석 → 목표 실루엣 오버레이 표시 → 사용자가 오버레이에 맞춰 직접 촬영 → 레퍼런스와 나란히 비교 → 레퍼런스 색감 적용

**J3. 망한 사진 복구 사용자:** 기존 사진 선택 → 문제 자동 분석·표시("기울기, 배경 방해 요소, 어두운 얼굴") → 스타일 선택 → 기본 보정 + 생성형 보완 순차 실행 → 자연스러운 버전과 적극 수정 버전 함께 제시 → 선택 또는 부분 되돌리기

## 4. 범위

### 4.1 In Scope (7일 내 구현)

전체 기능을 구현하되, 우선순위는 P0(끝까지 지킨다) > P1(기본 포함) > P2(밀리면 자른다)로 관리한다. 상세는 5장.

### 4.2 Out of Scope (프로토타입에서 제외)

- 자체 포즈 모델·자체 생성형 모델 학습 (검증된 사전학습 모델·API 조합 사용)
- iOS 완전 최적화 (Android 우선, iOS는 Day 7 여유 시 빌드만)
- 회원가입·로그인 (디바이스 UUID로 대체)
- 얼굴 형태·신체 비율·나이·인종적 특징 변경, 타인 얼굴 합성 (정책상 영구 제외)
- 상용 수준 대규모 개인화 모델, 결제, 푸시 알림, 소셜 공유 연동

### 4.3 컷라인 (일정 지연 시 자르는 순서)

1. iOS 빌드 → 2. IC-Light 릴라이팅(색보정으로 대체) → 3. 2인 포즈 템플릿 → 4. 스타일 탐색 카테고리(단일 리스트로) → 5. 암묵 신호 로깅 → 6. 구도 제안 후보 전환(스와이프) — 최후의 컷은 장면 스캔 제안(정적 프리셋 오버레이로 대체)

*주: 텍스트 안내·자동 촬영·일치도 게이지는 컷이 아니라 제품 결정으로 제외됨(가이드는 시각 오버레이만, 촬영은 사용자 수동).*

|---|---|---|

| F3-2 | 오버레이 안정화: 스트리밍 추적, 이동평균, 신뢰도 임계값, 히스테리시스 — 오버레이가 프레임마다 튀지 않는다 | P0 |
| F3-3 | 오버레이 구성: 추천 프레임(브래킷) + 피사체 배치 실루엣 + 수평선 + 정렬 대상 라인 하이라이트. 텍스트 지시·화살표·게이지는 표시하지 않음 | P0 |
| F3-4 | 일치도 점수는 UI에 표시하지 않고 세션 로그(final_match_score)로만 내부 기록 — KPI 측정용 | P1 |
| F3-6 | 장면 분류(인물 수, 전신/상반신/얼굴, 음식·건물·풍경·제품), 배경 복잡도, 역광·저조도·과노출 감지 | P1 |

**수용 기준:** 프리뷰가 멈추지 않는다(분석이 느려도 최신 프레임 유지). 안내가 프레임마다 바뀌지 않는다. 감지 API는 교체 가능한 추상화 계층 뒤에 둔다.

## 6. 비기능 요구사항

### 6.1 개인정보 및 안전

- **온디바이스 우선 처리:** 얼굴·인물 감지, 포즈 추정, 실시간 구도 계산, 기본 색감 보정, 스타일 추천, 기본 배경 분리
- **서버 업로드 필요 기능:** 고품질 생성형 제거, 아웃페인팅, 조명 재구성. 업로드 전 어떤 처리를 위해 전송되는지 표시
- **보관 정책:** 원본 자동 보관 금지, 편집 완료 후 서버 원본 즉시 삭제, 명시 저장 결과만 유지, 학습 활용 기본 비활성화, 위치·얼굴 정보 영구 저장 금지
- **생성형 표시:** 편집 이력에서 생성형 수정 여부 확인 가능, 결과 화면 뱃지

### 6.2 기술 아키텍처 (착수 결정 문서 반영)

```
[모바일 앱 — Kotlin + Jetpack Compose, Android 단일]
 ├─ 카메라: CameraX Preview·ImageAnalysis·ImageCapture
 ├─ 감지: ML Kit(포즈·얼굴), MediaPipe(분할) — 추상화 계층 뒤
 ├─ 센서: 자이로스코프·기울기
 ├─ 로컬 보정: RenderEffect·AGSL 또는 OpenCV
 └─ 로컬 저장: Room(SQLite) / 파일
        │ REST (업로드 → job → 폴링)
        ▼
[백엔드 — FastAPI 단일 서버]
 ├─ 프리셋(JSON)·레퍼런스 분석·편집 job 관리
 ├─ SQLite + 로컬 디스크 저장
 └─ 사용자 식별: 디바이스 UUID (로그인 없음)
        │
        ▼
[생성형 처리 — 자체 GPU + ComfyUI (headless API)]
 ├─ 객체 제거: LaMa (경량·고속)
 ├─ 고품질 inpaint/outpaint: FLUX.1 Fill [dev] (비상업 라이선스 — 상용화 시 교체)
 ├─ 조명: IC-Light (P2)
 └─ 검증: 얼굴 임베딩 비교(InsightFace/ArcFace)
```

*Spring Boot·MySQL·Redis·S3·WebSocket은 프로토타입 범위에서 제외(제안서 대비 축소). 확장 시점에 재도입 검토.*

### 6.3 데이터 모델 (핵심 엔티티)

 - **DeviceSettings:** 디바이스 UUID, 설정, 개인정보 동의, 기본 보정 강도 (앱 로컬)
- **StyleProfile:** 구도 취향, 색감 취향, 피사체 유형별 선호, 자주 쓰는 화면 비율
- **StylePreset:** 이름, 대표 이미지, 구도 파라미터, 색상 파라미터, 생성형 편집 정책
- **ReferenceImage:** 소유자, 위치, 임베딩, 인물·포즈 정보, 색상 팔레트, 화면 구조
- **ShootingSession:** 선택 스타일, 장면 유형, 촬영 시간, 안내 기록, 최종 일치도
- **Capture:** 원본, 분석 결과, 촬영 조건, 선택 결과
- **EditJob:** 요청 보정, 작업 상태, 입력 이미지, 결과 후보, 실패 원인
- **Feedback:** 결과 만족도, 저장 여부, 선호 결과, 부분별 피드백
