## 0. 문서 사용법

- 기능은 **모듈(M1~M15) → 기능(M4-03) → 세부 기능(체크박스)** 3단계로 정의한다. 체크박스가 곧 구현 체크리스트다.
- 각 기능 헤더 표기: `[우선순위 | 담당 | 목표 Day | 의존성]`
  - 우선순위: **P0**(끝까지 지킨다) / **P1**(기본 포함) / **P2**(컷라인 대상)
  - 담당: **A**(모바일·실시간) / **B**(백엔드·생성형·스타일) / **AB**(공동)
- **완료 조건(DoD)** 을 만족해야 완료 처리한다. 부분 구현은 완료가 아니다.
- PRD 요구사항과의 대응은 12장 추적 매트릭스 참조.

## M2. 온보딩·취향 선택 `[P0 | A(UI)+B(로직) | Day 5]`

### M2-01. 온보딩 플로우 `[P0 | A | Day 5 | 의존: M1-02]`

- [ ] 화면 순서: 서비스 소개(1) → 사진 권한 → 카메라 권한 → 취향 카드 선택 → 스타일 프로필 결과 → 개인정보 처리 안내
- [ ] 소개 화면: 슬로건 "원하는 무드로 찍고, 원하는 무드로 살린다" + 3단계 가치(취향 발견→촬영 지원→결과 복구) 1화면 요약
- [ ] 개인정보 안내: 온디바이스 처리 원칙, 서버 전송 시점 고지, 원본 미보관 정책 표시
- [ ] 온보딩 완료 플래그 저장, 재실행 시 홈 직행

**DoD:** 신규 설치 → 첫 촬영 화면 도달까지 60초 이내(내부 테스트 기준).

### M2-02. 취향 카드 선택 `[P0 | A | Day 5 | 의존: 카드 에셋]`

- [ ] 카드 20장 그리드 표시(2열), 탭으로 선택/해제, 선택 수 카운터
- [ ] 최소 5장 선택 시 "다음" 활성화, 상한 없음
- [ ] 카드 에셋: Unsplash/Pexels 라이선스 확인본, 8개 차원(근접/전신, 중앙/비대칭, 밝음/어두움, 배경/인물, 선명/흐림, 따뜻/차가움, 정돈/스냅) 각 차원당 양극 2~3장씩 커버
- [ ] 각 카드에 사전 계산된 특성 벡터 메타데이터(JSON) 내장 — 런타임 이미지 분석 없음

**DoD:** 20장 모두 특성 메타데이터를 갖고 있고, 선택 결과가 M2-03에 전달된다.

### M2-03. 초기 스타일 프로필 생성 `[P0 | B(로직)+A(통합) | Day 4 | 의존: M2-02 메타데이터]`

**온디바이스 실행** — 카드 특성 벡터가 앱 번들(cards.json)에 있으므로 서버 호출 없이 기기에서 계산하고 로컬 style_profile에 저장한다. B가 로직 모듈(TS)을 작성, A가 앱에 통합.

- [ ] 입력: 선택 카드들의 특성 벡터 / 출력: StyleProfile(구도 취향 + 색감 취향 분리) → 로컬 저장
- [ ] 알고리즘: 선택 카드 벡터의 가중 평균 + 분산 기반 확신도. 상충 차원(예: 밝음과 어두움 동시 선택)은 확신도 낮음으로 기록해 추천 다양성에 반영
- [ ] 프로필 요약 문구 생성: 특성값 → 일상 언어 템플릿 매핑("밝은 자연광, 넓은 배경, 자연스러운 인물 배치") — 전문 용어 금지
- [ ] 추천 스타일 3개 산출: 프리셋 6종과 프로필 벡터의 거리 계산, 상위 3개
- [ ] 서로 다른 선택 → 서로 다른 요약·추천이 나오는지 고정 테스트 케이스 2세트 작성 (시연 C 대비)

**DoD:** 테스트 케이스 2세트가 서로 다른 프로필 요약과 추천 3종을 반환한다.

## M3. 스타일 시스템 `[P0 | B | Day 3]`

### M3-01. StylePreset 스키마·프리셋 6종 `[P0 | B | Day 3 | 의존 없음]`

- [ ] JSON 스키마 확정(인터페이스 계약 Day 1 고정):

```json
{
  "id": "clean_social",
  "name": "Clean Social",
  "displayName": "깔끔한 소셜",
  "thumbnail": "presets/clean_social.jpg",
  "composition": {
    "targetAspectRatio": "4:5",
    "subjectScaleRange": [0.35, 0.55],
    "subjectPosition": "center|third_left|third_right",
    "headroomRange": [0.05, 0.12],
    "horizonPosition": 0.5,
    "cameraPitchRange": [-5, 5],
    "posePattern": "natural_standing",
    "backgroundRatio": [0.4, 0.6]
  },
  "color": {
    "colorTemperature": 5800,
    "exposureBias": 0.3,
    "contrast": 0.1,
    "saturation": 0.05,
    "grain": 0.0,
    "vignette": 0.0,
    "blurStrength": 0.0,
    "fade": 0.0
  },
  "cropFreedom": 0.2,
  "generativeEditPolicy": "allow_background|allow_extend|conservative"
}
```

- [ ] 6종 초기값 작성: Clean Social / Candid Feed / Bright Review / Soft Film / Casual Portrait / Night Street (특징은 PRD F2-1)
- [ ] 구도 정보(composition)와 색상 정보(color)는 **분리 저장** — 가이드는 composition만, 보정은 color만 소비
- [ ] 프리셋은 서버 JSON 파일로 관리, 앱은 기동 시 fetch + 로컬 캐시 (재배포 없이 튜닝 가능)
- [ ] 매일 저녁 AB 공동 튜닝 세션에서 실사진으로 값 조정 → git 커밋 이력으로 버전 관리

**DoD:** 6종 JSON이 스키마 검증을 통과하고 앱에서 로드된다.

### M3-02. 스타일 적용 해석기(Resolver) `[P0 | B(로직 스펙)+A(앱 구현) | Day 3]`

- [ ] 입력: StylePreset + StyleProfile + 장면 유형 / 출력: 최종 목표 파라미터(ResolvedStyle)
- [ ] 결합 규칙: `최종값 = 프리셋 기본값 + 개인 프로필 보정치(확신도 가중)` — 같은 프리셋도 사용자마다 다른 결과
- [ ] 범위형 파라미터(subjectScaleRange 등)는 프로필 선호값이 범위 내면 선호값, 밖이면 가까운 경계로 클램프
- [ ] 스타일 강도 슬라이더(0~100%): color 파라미터에만 선형 스케일 적용 `[P1]`

**DoD:** 동일 프리셋 + 상이한 프로필 2개 → 상이한 ResolvedStyle이 산출되는 단위 테스트 통과.

### M3-03. 개인 프리셋·혼합 `[P1 | B | Day 5]`

- [ ] 현재 ResolvedStyle을 이름 지정해 개인 프리셋으로 저장
- [ ] 두 스타일 혼합: 파라미터별 선형 보간(비율 슬라이더)
- [ ] 개인 프리셋 목록·삭제

**DoD:** 저장한 개인 프리셋으로 촬영 시 동일 파라미터가 재현된다.

### M4-04. 오버레이 렌더링 엔진 `[P0 | A | Day 2]`

- [ ] Skia/Compose Canvas 레이어: 목표 영역(반투명 브래킷), 실루엣, 수평선·정렬 라인을 프리뷰 위에 렌더 — 텍스트·화살표·게이지 레이어 없음
- [ ] 좌표계 통일: 감지 결과(정규화)→화면 좌표 변환 유틸
- [ ] 레퍼런스 원본 반투명 오버레이(투명도 슬라이더 0~60%)
- [ ] 오버레이 갱신은 애니메이션 보간(급격한 점프 방지)

**DoD:** 감지 박스·목표 영역·텍스트가 프리뷰 위에 정렬 오차 없이 표시된다.

## M6. 실시간 촬영 가이드 `[P0 | A | Day 3]`

### M6-01. 장면 스캔 & 최적 구도 제안 `[P0 | A+B | Day 3 | 의존: M3-02, M5-03, M7-02]`

**제안형 시각 가이드 (영뽀또식):** 앱이 현재 장면을 분석해 "이렇게 찍으세요"를 화면에 오버레이로 그려준다. **안내는 여기서 끝이다** — 텍스트 지시·화살표·일치도 게이지·자동 촬영 없이, 사용자가 오버레이를 보고 스스로 맞춰 직접 셔터를 누른다.

- [ ] **장면 스캔**: 카메라 기동 시·장면 대폭 전환 시(전역 움직임 감지 후 정지) 1~2초 스캔 → FrameFeatures + 장면 구조(리딩 라인, 개방 공간, 수평선, 밝기 분포)
- [ ] **구도 후보 생성 — 규칙 라이브러리 × 장면 구조**: 각 규칙을 현재 장면에 적용해 실현 가능한 프레이밍 후보를 만든다
  - 핵심 3규칙(Day 3, P0): 삼분할 배치(개방 공간에 피사체 지점 산출), 수평선·주요 라인 정렬, 여백 균형(스타일의 backgroundRatio·headroom 반영)
  - 확장 3규칙(P1): 리딩 라인 수렴점 활용, 중앙 대칭, 프레임 속 프레임(문·창 구조)
- [ ] **후보 점수화**: `구도 원칙 점수 × 사용자 프로필 적합도(ResolvedStyle.composition) × 실현 가능성(요구 이동량이 적을수록 가점)` — 같은 장소라도 사용자마다 다른 구도가 제안된다(시연 C 근거)
- [ ] 상위 1개 기본 제안, **좌우 스와이프로 후보 2~3개 전환** `[P1]`, 재스캔 버튼
- [ ] 출력(TargetComposition): 추천 프레임, 피사체 배치 지점(발 위치·상체 영역), 목표 여백·수평·비율, 목표 포즈(선택적)
- [ ] 목표 소스 우선순위: 레퍼런스 선택 시 ① 레퍼런스(M7-02) / 스타일 모드 ② 장면 스캔 제안 × 프리셋 / 자유 모드 ③ 장면 스캔 제안 × 프로필
- [ ] 장면이 크게 바뀌면 자동 재제안(히스테리시스 적용 — 사소한 움직임에는 목표 고정), 소스 간 전환 시 오버레이 부드럽게 갱신

**DoD:** 카메라를 들고 2초 이내에 추천 구도 오버레이가 나타난다. 두 개의 다른 프로필로 같은 장소를 비추면 다른 구도가 제안된다.

### M6-02. 오버레이 안정화 `[P0 | A | Day 3]`

- [ ] 오버레이 좌표에 이동평균(윈도 5프레임) 적용 — 프레임마다 튀지 않는다
- [ ] 신뢰도 임계값 미달 감지 결과는 오버레이 갱신에 사용하지 않음(마지막 안정값 유지)
- [ ] 히스테리시스: 목표 재계산(재제안)은 장면이 크게 바뀌었을 때만 — 사소한 움직임에는 목표 고정
- [ ] 감지가 계속 불안정하면 오버레이를 잠시 숨김(깜빡임보다 숨김이 낫다)

### M6-04. 가이드 오버레이 UI `[P0 | A | Day 3 | 의존: M4-04]`

**"찍는 순간 화면에 구도가 직접 나타난다"** — 스캔 완료 즉시 추천 구도가 오버레이로 뜬다. 맞추는 판단과 셔터는 사용자 몫이다.

- [ ] **추천 구도 오버레이**: 추천 프레임(모서리 브래킷), 피사체 배치 지점 고스트 마커(인물 실루엣 + 발 위치 표시), 정렬 대상 라인(수평선·리딩 라인) 하이라이트
- [ ] **표시하지 않는 것(제품 결정):** 텍스트 지시 문구, 방향 화살표, 일치도 게이지, 자동 촬영 관련 UI 일체
- [ ] 오버레이 표시 on/off 토글(상단) — 오버레이 없이 찍고 싶은 사용자 존중
- [ ] 후보 전환 시(스와이프) 오버레이 애니메이션 전환 `[P1]`

**M6 DoD:** 고정 시나리오(기울임, 좌측 치우침)에서 추천 구도 오버레이가 안정적으로 표시되고 프레임 간 깜빡임이 없다. 오버레이는 촬영된 사진에 포함되지 않는다.

## 10. API 명세 (인터페이스 계약)

공통: Base URL `http://{server}/api/v1`, 헤더 `X-Device-Id: {uuid}`, 에러 `{code, message, retryable}`

**서버 API는 아래 4개가 전부다.** 온보딩 프로필 생성·피드백 반영·추천은 온디바이스 로직(M2-03, M14-02)이며 서버 엔드포인트가 없다. 서버는 사용자 상태를 저장하지 않는다(DB 스키마 v2.0).

### 10.1 `GET /presets`

프리셋 6종 배열(M3-01 스키마)의 정적 JSON 서빙. `ETag` 지원으로 앱 캐시. 앱 번들에 동일 파일을 내장해 서버 미접속 시 폴백.

### 10.2 `POST /references/analyze` (multipart)

```json
// Response 200 (동기, 목표 5초 이내)
{
  "analysis": {
    "peopleCount": 2,
    "subjects": [{ "bbox": [0.2,0.3,0.3,0.6], "faceSize": 0.08, "pose": {...} }],
    "cameraHeight": "chest_level",
    "horizon": 0.52, "tilt": -1.2,
    "backgroundRatio": 0.55, "aspectRatio": "4:5",
    "palette": ["#E8D5C0", "..."], "colorTemperature": 5200,
    "luminanceHistogram": [...]
  },
  "targetComposition": {...},   // M6-01 소비용
  "colorTarget": {...}          // M7-04 소비용
}
```

응답은 **무저장** — 서버는 응답 직후 업로드 파일을 삭제하고, 클라이언트가 `cached_references`(content_hash 키)에 보관한다.

### 10.3 `POST /edit-jobs`

```json
// Request — jobId·captureId는 클라이언트가 생성(로컬-서버 ID 공유)
{
  "jobId": "job_456",
  "captureRef": "cap_123",
  "image": "(multipart)",
  "operations": [
    { "type": "remove_objects", "masks": [...], "auto": true },
    { "type": "outpaint", "direction": "top", "amountRatio": 0.15 },
```

## 13. 공통 예외 처리 규칙

1. **네트워크 단절:** 로컬 기능(촬영·가이드·기본 보정)은 전부 동작 유지. 생성형 메뉴만 비활성 + "연결 후 이용 가능" 표시. job 요청 실패는 로컬 큐 저장 후 재시도 1회.
2. **감지 실패(인물 미검출):** 인물 기반 오버레이(실루엣·배치 마커)를 숨기고 수평선·프레임 가이드만 유지. 상태 문구 1줄("인물이 화면에 들어오면 구도를 맞춰드릴게요")은 허용 — 행동 지시가 아닌 상태 표시.
3. **생성 실패/검증 탈락:** 사용자에게 실패를 노출하지 않고 기본 보정 결과 + "자연스러운 보정만 적용했어요" 문구로 폴백.
4. **저조도·극단 조명:** "조명이 어두워요" 상태 칩만 표시(행동 지시 아님), 오버레이는 유지.
5. **서버 job 5분 초과:** 타임아웃 → failed 처리 → 클라이언트 폴백 루틴.
6. **저장 공간 부족:** 촬영 전 감지 시 경고, 결과 저장 실패 시 원본만이라도 보존.
