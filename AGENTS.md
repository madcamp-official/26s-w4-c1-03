# AGENTS.md — 감도 (GAMDO) 프로젝트 컨텍스트

> **이 파일의 목적:** 새 세션(사람·AI 에이전트 무관)이 이 파일 하나만 읽고 프로젝트를 정확히 이해하고 작업에 착수할 수 있게 한다.
> **읽는 규칙:** 이 파일의 "확정 결정"은 다른 모든 문서보다 우선한다. 문서 간 충돌을 발견하면 §6의 우선순위 표를 따르고, 오래된 서술을 이 파일 기준으로 고쳐라.

---

## 1. 프로젝트 한 줄 요약

**감도(GAMDO)** — 사용자가 좋아하는 사진을 고르면, 그 느낌으로 찍는 법을 촬영 화면의 **시각 오버레이**로 보여주고, 찍힌 결과를 같은 감성으로 자동 보정·생성 복구하는 **개인화 카메라 Android 앱**. (코드네임: MOODFRAME)

- 슬로건: "원하는 무드로 찍고, 원하는 무드로 살린다"
- 정체성: 사진을 대신 만들어주는 AI가 아니라, **실제로 찍은 사진을 중심에 두는 카메라 도구**
- 핵심 차별점: 촬영 전(Pre-capture 가이드)과 촬영 후(Post-capture 복구)를 하나의 흐름으로 연결. 기준은 "좋은 사진"이 아니라 **"사용자가 좋아하는 사진과 비슷한가"**

## 2. 팀·기간·목표

- **2인 개발, 7일 프로토타입.** 담당 A(모바일·카메라), 담당 B(백엔드·CV·생성 + 온디바이스 로직 모듈 작성)
- 목표는 상용 베타가 아니라 **데모 시나리오 2개를 실기기에서 끊김 없이 시연**:
  - 시나리오 A: 레퍼런스/스타일 따라 찍기 (분석 → 오버레이 표시 → 사용자가 맞춰 수동 촬영 → 색감 적용 → 비교)
  - 시나리오 B: 망한 사진 살리기 (진단 → 기본·스타일 보정 → 생성형 객체 제거/여백 확장 → 슬라이더 비교)

## 3. 확정 결정 (이것이 최종 — 다른 문서의 오래된 서술보다 우선)

| # | 결정 | 내용 | 비고 |
|---|---|---|---|
| D1 | 플랫폼 | **Android 단일, Kotlin 네이티브 + Jetpack Compose** | 초기 RN 검토는 폐기됨. iOS 없음 |
| D2 | 가이드 방식 | **시각 오버레이 전용**: 추천 프레임(브래킷) + 인물 배치 실루엣 + 수평선만 표시. **텍스트 지시 문구·방향 화살표·일치도 게이지·자동 촬영은 만들지 않는다** (컷이 아니라 제품 결정). 촬영은 사용자가 수동 셔터로. 인물이 목표 프레임에 들어오면 오버레이 색 전환(흰→민트)이 유일한 피드백 | 가장 최근 결정. 구버전 문서에 "단일 안내 문구", "자동 촬영"이 남아 있으면 이 결정이 우선 |
| D3 | 구도 제안 | 영뽀또식 **제안형**: 카메라를 들면 장면(리딩 라인·개방 공간·수평선)을 스캔해 최적 구도를 오버레이로 먼저 제안. 규칙 3개(삼분할 배치, 수평선 정렬, 여백 균형) × 사용자 프로필 점수화 | 불안정 시 정적 프리셋 프레임으로 다운그레이드 |
| D4 | 계정·데이터 | **로그인 없음.** 디바이스 UUID만. **선호도·프로필·피드백·세션 기록은 전부 앱 로컬(Room) 저장** — 서버는 사용자를 저장하지 않는 무상태 처리기 | 앱 삭제 = 데이터 소실(온보딩에 고지) |
| D5 | 서버 | FastAPI + SQLite(작업 추적 3테이블: edit_jobs, edit_job_files, schema_migrations) + 로컬 디스크 + DB 폴링 워커. **API 4개뿐**: `GET /presets`(정적 JSON), `POST /references/analyze`(무저장, 응답 후 즉시 파일 삭제), `POST /edit-jobs`, `GET /edit-jobs/{id}`(1초 폴링) | PostgreSQL·Redis·S3·로그인 API는 확장 시. 온보딩 프로필 생성·피드백 반영은 온디바이스 로직 |
| D6 | 스타일 | 프로토타입은 **3종만**: Clean Social, Soft Film, Bright Review. StylePreset은 composition(구도)과 color(보정) 분리 구조의 JSON, 서버 정적 파일 + 앱 번들 폴백 | PRD의 6종 서술은 정식 버전 기준 |
| D7 | 생성형 | 자체 GPU + ComfyUI headless. **객체 제거(LaMa) 1순위**, 큰 영역·여백 확장은 FLUX.1 Fill [dev](비상업 라이선스 — 상용화 시 교체). `GenerativeEditProvider` 인터페이스로 공급자 교체 가능. Day 1에 Gemini 무료 티어와 품질 비교 후 확정 | 명시적 실행 시에만 서버 업로드 |
| D8 | 안전·개인정보 | 얼굴·신체·나이·인종 변경 기능 **자체를 만들지 않음**. 생성 결과는 얼굴 임베딩(InsightFace) 전후 비교로 검증, 실패 시 기본 보정 폴백(사용자에게 실패 미노출). 원본 비파괴, 서버 업로드분 즉시 삭제, EXIF 위치 스트립, 임베딩 저장 금지 | 스키마 차원에서도 강제됨 |
| D9 | 촬영 범위 | 1인 인물 우선, 비율 4:5·1:1만 | 2인·음식·풍경 특화는 후속 |
| D10 | 브랜드 | 이름 "감도"(ISO 감도 + 감성의 정도). 다크 차콜 UI, 사진이 주인공, 전문 용어 금지("삼분할" 대신 일상 언어), 영뽀또의 정보 구조는 참고하되 시각 표현(색·아이콘·서체)은 독자 설계 | |

## 4. 아키텍처 요약

```text
[Android 앱 — Kotlin/Compose]                [서버 — FastAPI+SQLite]      [GPU — ComfyUI]
 CameraX(Preview/Analysis/Capture)            edit_jobs 큐·폴링    ──────→  LaMa 객체 제거
 ML Kit(얼굴·포즈) + MediaPipe(분할)          /references/analyze(무저장)   FLUX.1 Fill 확장
 센서(수평·흔들림)                            /presets 정적 서빙            InsightFace 검증
 AlignmentEngine(오버레이 좌표+안정화)
 LocalEditor(수평→노출→스타일 LUT, 2초 내)
 Room(로컬 14테이블 = 데이터 원천)
```

- 온디바이스: 감지, 오버레이, 기본 보정, 프로필 생성·피드백 반영, 추천 (오프라인 전부 동작)
- 서버 필요: 레퍼런스 분석, 생성형 편집만
- B가 작성해 A가 통합하는 순수 Kotlin 모듈 4개(JVM 단위 테스트 필수): `FrameFeatureCalculator`, `AlignmentEngine`, `ProblemDiagnoser`, `ProfileEngine`
- 임계값·가중치는 전부 `guide_config.json` 외부화(리허설 현장 튜닝용)

## 7. 불변 규칙 (작업 중 위반 금지)

1. **완료(Done)의 정의**: 코드 작성이 아니라 "사용자가 실기기에서 해당 흐름을 끝까지 수행 가능" + 모듈은 단위 테스트 통과.
2. **DB 스키마 동결**: 컬럼 변경·삭제 금지, 추가(additive)만 허용. 파라미터류 구조 변경은 JSON payload `v` 증가로. 얼굴 임베딩·위치 정보 영구 저장 컬럼 추가 금지.
3. **D2·D8은 재론 불가**: 텍스트 안내/자동 촬영/게이지 추가 제안, 얼굴 변형 기능 제안은 사용자(제품 오너)의 명시적 번복 없이 구현하지 않는다.
4. **Day 6부터 신규 기능 금지** — 오류·성능·데모 품질만.
5. **생성 실패를 사용자에게 노출하지 않는다** — 항상 기본 보정 결과로 폴백("자연스러운 보정만 적용했어요").
6. **더미로 속이지 않는다** — 생성 API 불안정 시 고정 이미지를 결과로 위장하지 말고 기능 플래그로 숨긴다.
7. UI 문구에 전문 용어 금지(삼분할·헤드룸·네거티브 스페이스 등), 상태 표시는 허용하되 행동 지시 문구는 금지(D2).

## 9. 용어 사전

| 용어 | 의미 |
|---|---|
| StylePreset | 스타일 1개의 구도(composition)+보정(color) 파라미터 JSON |
| StyleProfile | 사용자 개인의 취향 벡터(구도/색감 분리, 차원별 확신도 포함, 로컬 저장) |
| ResolvedStyle | 프리셋 + 개인 프로필을 결합한 최종 목표 파라미터 |
| FrameFeatures | 프레임별 실시간 특징(인물 박스·여백·수평·밝기 등) |
| StyleTarget / TargetComposition | 오버레이가 그릴 목표 구도(프리셋 또는 레퍼런스 분석에서 생성) |
| OverlayState | AlignmentEngine 출력 — 목표 프레임·실루엣·수평선 좌표 + 표시 여부 |
| matchScore | 내부 일치도 점수 — **UI 표시 금지**, KPI 로그 전용 |
| Rescue / 사진 살리기 | 기존 사진 진단→보정→생성 복구 흐름 |
| EditJob | 서버 비동기 생성 편집 작업(클라이언트가 ID 생성, 1초 폴링) |
